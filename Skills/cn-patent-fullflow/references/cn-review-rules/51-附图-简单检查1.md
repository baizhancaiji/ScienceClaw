## 审查规则

### 规则1：摘要附图与说明书附图的依赖关系
如果说明书附图章节中没有图片，那么摘要附图章节中也不得有图片；但如果说明书附图章节中有图片，那么摘要附图章节中必须有1张图片。
- 检查方法：从 image_analysis_<timestamp>.json 中分别获取"说明书附图"章节和"摘要附图"章节的图片数量
- 如果说明书附图数量 = 0 且 摘要附图数量 > 0 → 问题：说明书没有附图但摘要有附图
- 如果说明书附图数量 > 0 且 摘要附图数量 = 0 → 问题：说明书有附图但摘要缺少附图
- 如果说明书附图数量 = 0 且 摘要附图数量 = 0 → 符合要求（发明专利可以无附图）
- 如果说明书附图数量 > 0 且 摘要附图数量 = 1 → 符合要求
- 如果说明书附图数量 > 0 且 摘要附图数量 > 1 → 违反规则2，按规则2处理

### 规则2：摘要附图数量限制
如果说明书附图章节中有图片，那么摘要附图章节中只能有1张图片。
- 如果说明书附图数量 > 0 且 摘要附图数量 < 1（即0张）→ 提醒用户从说明书附图中引用其中1张图作为摘要附图（此问题也与规则1重叠，优先以规则1的问题描述为准）
- 如果说明书附图数量 > 0 且 摘要附图数量 > 1 → 提醒用户从说明书附图中引用1张最能说明该发明技术方案主要技术特征的附图作为摘要附图，并删除其余图片

### 规则3：摘要附图来源约束
摘要附图的图片，必须是引用自说明书附图的其中一张图。如果摘要附图不是说明书附图章节中的任一图片，应当提醒用户摘要附图只能从说明书附图中选择1张图，不得使用其它图片。
- 检查方法：从 image_analysis_<timestamp>.json 中获取摘要附图的 source_file（源文件名/哈希），确认该源文件也出现在说明书附图的图片列表中
- 如果摘要附图的源文件不在说明书附图的源文件列表中 → 问题：摘要附图不是来自说明书附图

### 规则4：附图标记编号与一致性（委托给图片审查子Agent）
说明书附图章节中的图片的附图标记应当使用阿拉伯数字编号。说明书文字部分中未提及的附图标记不得在附图中出现，附图中未出现的附图标记也不得在说明书文字部分中提及。
- ⚠️ **此规则的图片级检查部分委托给图片审查子Agent执行**
- 本Agent负责的元数据级前置检查：从 section_description_<timestamp>.json（说明书文本）中提取所有附图标记（如"图1"、"图2"、"(1)"、"（1）"等模式），整理为附图标记列表，传递给每个图片审查子Agent供其对比使用
- 图片审查子Agent会逐张查看每张说明书附图，检查其中的附图标记是否为阿拉伯数字编号，并与本Agent提供的附图标记列表进行交叉验证

### 规则5：专利类型与附图要求
发明专利申请可以没有附图，但实用新型专利申请必须有附图。如果申请类型为实用新型但说明书附图章节中没有图片，应提醒用户补充附图。
- 检查方法：从 section_description_<timestamp>.json（说明书文本）中判断专利申请类型
  - 如果标题或正文中出现"实用新型"字样 → 判定为实用新型专利申请
  - 如果标题或正文中仅出现"发明"字样 → 判定为发明专利申请
  - 如果两者均未明确出现或无法判断 → 在审查意见中以 comment 类型说明无法确定申请类型
- 如果判定为实用新型 且 说明书附图数量 = 0 → 问题：实用新型专利申请必须有附图
- 如果判定为发明 → 无论是否有附图均符合要求

### 规则6：附图注释规范（委托给图片审查子Agent）
说明书附图中除了必需的词语外，不应当含有其他的注释；但对于流程图、框图一类的附图，应当在其框内给出必要的文字或符号。
- ⚠️ **此规则完全委托给图片审查子Agent执行**——需要查看每张图片的实际视觉内容才能判断
- 图片审查子Agent会逐张查看每张说明书附图，检查是否存在不当注释，以及流程图/框图是否有必要的文字或符号

### 规则7：说明书附图不得有重复图片
说明书附图章节中不得有重复的图片，即同一张图片（以 source_file 源文件名/哈希为判定依据）最多在摘要附图章节中出现1次、在说明书附图章节中出现1次。如果同一 source_file 在说明书附图章节中出现多次，则为重复图片。
- 检查方法：从 image_analysis_<timestamp>.json 中获取"说明书附图"章节的图片列表，统计每个 source_file 的出现次数
- 如果某个 source_file 在说明书附图章节中出现次数 > 1 → 问题：说明书附图中存在重复图片
- 同时检查：如果某个 source_file 在摘要附图章节中出现次数 > 1 → 问题：摘要附图中存在重复图片
- 如果某个 source_file 同时出现在摘要附图和说明书附图中 → 符合要求（摘要附图可以引用说明书附图中的图片）

### 规则8：摘要附图章节无文字时的处理
如果摘要附图章节中没有文字内容，则应当添加批注提醒用户。批注方式为：将批注定位到文档页眉中的"摘要附图"文本上，在批注内容中说明摘要附图章节缺少文字说明。
- 检查方法：从 image_analysis_<timestamp>.json 中获取"摘要附图"章节的信息，或从 section_description_<timestamp>.json 的 extracted_text 中检查摘要附图章节是否包含文字
- 如果摘要附图章节有图片但无任何文字 → 问题：摘要附图章节缺少文字说明
- 输出格式特殊处理：
  - context 字段：设为 "摘要附图"（文档页眉中的章节标题文本）
  - highlight_text 字段：设为 "摘要附图"
  - action_type：comment
  - issue：说明摘要附图章节缺少文字说明
  - suggestion：建议在摘要附图章节添加对摘要附图的简要说明文字

### 规则9：说明书附图每幅图应有对应名称
说明书附图章节中的每一幅图片都应有对应的名称，该名称为图片下方的文字。例如，说明书附图章节中 image2 下方的文字为"图1"，则 image2 即为说明书附图1。
- 检查方法：从 image_analysis_<timestamp>.json 中获取"说明书附图"章节的每张图片信息，检查每张图片下方是否有对应的文字标注（如"图1"、"图2"等）
- 如果某张图片下方没有任何文字 → 问题：该说明书附图缺少名称标注
- 如果某张图片下方有文字但不符合"图N"的格式 → 问题：该说明书附图的名称标注格式不规范
- 正确格式示例："图1"、"图2"、"图3"等
- 错误格式示例：无文字、仅有数字无"图"字前缀、使用非阿拉伯数字等

### 规则10：说明书附图应当顺序编号
说明书附图应当按照阿拉伯数字顺序编号，不得乱序或跳序编号。即如果说明书附图共有N幅图，则其编号应当依次为图1、图2、……、图N，不得出现编号跳跃（如从图1直接跳到图3，缺少图2）或编号乱序（如图3出现在图1之前）的情况。
- 检查方法：
  1. 从 image_analysis_<timestamp>.json 和 section_description_<timestamp>.json 中提取说明书附图章节中所有图片的名称/编号
  2. 提取所有"图N"中的数字N，形成编号列表
  3. 检查编号是否从1开始连续递增
- 如果编号不从1开始 → 问题：说明书附图编号应从图1开始
- 如果编号存在跳跃（如1、2、4，缺少3） → 问题：说明书附图编号不连续，缺少图N
- 如果编号乱序（如2、1、3，图2出现在图1之前） → 问题：说明书附图编号顺序混乱，图N应排在图M之后
- 如果编号有重复（如出现两个图1） → 问题：说明书附图编号重复（与规则7的图片重复不同，此处指编号重复）

### 规则11：附图说明与说明书附图内容的对应性（委托给图片审查子Agent）
说明书中对附图的说明（即附图说明章节中的文字描述，如"图1是XXX的结构示意图"）应当与说明书附图中图片的实际内容相对应。如果附图说明描述的内容与图片实际展示的内容不一致，则应当指出。
- ⚠️ **此规则的图片级检查部分委托给图片审查子Agent执行**
- 本Agent负责的元数据级前置检查：
  1. 从 section_description_<timestamp>.json（说明书文本）中提取附图说明章节的内容，解析出每幅图的说明文字（如"图1是XXX的结构示意图"、"图2是XXX的剖视图"等）
  2. 将每幅图的说明文字整理为列表，传递给每个图片审查子Agent供其对比使用
  3. 检查附图说明中提及的图号是否与说明书附图章节中实际存在的图号一致：
     - 如果附图说明中提及了某个图号但说明书附图中不存在该图 → 问题：附图说明中提及了不存在的附图
     - 如果说明书附图中存在某个图号但附图说明中未提及 → 问题：说明书附图存在但附图说明中未说明
- 图片审查子Agent会逐张查看每张说明书附图，将图片的实际视觉内容与附图说明中的文字描述进行对比，检查是否一致

### 规则12：说明书附图章节无锚定文本时的处理
当为说明书附图章节添加批注时，若未提供或检测不到明确的锚定文本（如说明书附图章节中没有文字内容，或文字内容无法作为有效的锚定点），应自动使用页眉中的"说明书附图"章节标题作为批注文本内容，批注定位点必须精确设置为该页眉区域。
- 检查方法：从 image_analysis_<timestamp>.json 中获取"说明书附图"章节的信息，或从 section_description_<timestamp>.json / section_description_fig_<timestamp>.json 的 extracted_text 中检查说明书附图章节是否包含可作为锚定点的文字
- 如果说明书附图章节有图片但无任何可锚定的文字 → 使用页眉"说明书附图"作为锚定
- 输出格式特殊处理：
  - context 字段：设为 "说明书附图"（文档页眉中的章节标题文本）
  - highlight_text 字段：设为 "说明书附图"
  - action_type：comment
  - issue：说明说明书附图章节缺少可锚定的文字内容
  - suggestion：建议在说明书附图章节添加对附图的文字标注（如"图1"、"图2"等）

## 返回图片列表信息给主Agent

在完成规则1-3、5、7-10、12的检查后，你必须执行以下步骤（⛔禁止自行创建图片审查子Agent，由主Agent在第7批统一创建）：

1. 从 `image_analysis_<timestamp>.json` 中获取"说明书附图"章节的图片列表
2. 如果说明书附图数量 N > 0：
   a. 整理每张图片的以下信息，通过返回值传递给主Agent：
      - 图片文件的绝对路径（从 image_analysis JSON 的图片信息中获取实际文件路径，位于 `<work_dir>/images/` 目录下）
      - 该图片的逻辑名称/序号（如"说明书附图第1张图"）
      - 该图片的 fig_text_name（如"图1"、"图2"等）
      - 附图标记列表（你在规则4前置检查中从说明书文本提取的附图标记列表）
      - 附图说明文字列表（你在规则11前置检查中从说明书文本提取的每幅图的说明文字，如"图1是XXX的结构示意图"）
   b. 主Agent将根据你返回的信息，在第7批动态创建N个图片审查子Agent（每个仅读取1张图片），使用 `references/agent-fig-review.md` 配置，并行执行规则4、6、11
3. 如果说明书附图数量 N = 0：
   - 在返回值中明确告知主Agent图片数量为0
   - 规则4和规则6无图片可检查，跳过

### context 字段锚定规范（违反将导致批注添加必然跳过）
  - **核心原则**：context 必须是文档 extracted_text 中实际存在的文本（逐字 verbatim copy）
  - **禁止**使用 Agent 自行构造的描述性文本作为 context（以下均为错误示例）：
    - ❌ `"摘要附图图1（源文件：image1.png）"` — 文档中不存在此文本
    - ❌ `"说明书附图图1/图2/图3"` — 文档中不存在此文本
    - ❌ `"摘要附图章节包含1张图片"` — 这是概括性描述，非原文
  - **正确做法（按规则分类）**：
    - **规则1-3（摘要附图相关问题）**：从 extracted_text 中定位摘要附图章节的实际文本内容（通常为图号引用，如 `"图5"`），将其同时填入 context 和 highlight_text 字段
    - **规则3特殊说明（摘要附图来源异常）**：这是最高频的错误类型。正确做法是：在 extracted_text 中搜索摘要附图章节位置（通常在摘要正文之后、权利要求书之前），找到其中的实际文本（如 `"图5"` 或 `"图1"` 等），用该文本作为 context 和 highlight_text；完整的诊断信息放在 issue 字段中
    - **规则5（专利类型与附图要求）**：使用说明书中用于判断专利类型的依据句子（如包含"实用新型"字样的句子）作为 context
    - **规则7（说明书附图重复图片）**：从 extracted_text 中定位说明书附图章节的实际文本（通常为"图1"、"图2"等图号），选择重复图片对应的图号作为 context 和 highlight_text
    - **规则8（摘要附图章节无文字）**：从 extracted_text 中定位文档页眉中的"摘要附图"章节标题文本，将其同时作为 context 和 highlight_text
    - **规则9（说明书附图缺少名称）**：从 extracted_text 中定位说明书附图章节的实际文本，选择缺少名称的图片附近的图号或章节标题作为 context
    - **规则10（说明书附图编号问题）**：从 extracted_text 中定位说明书附图章节中编号有问题的图号文本（如"图3"），将其作为 context 和 highlight_text
    - **规则11元数据级（附图说明与附图对应性）**：从 extracted_text 中定位附图说明章节的实际文本（如"图1是XXX的结构示意图"），将其作为 context 和 highlight_text
    - **规则12（说明书附图章节无锚定文本）**：从 extracted_text 中定位文档页眉中的"说明书附图"章节标题文本，将其同时作为 context 和 highlight_text
    - **说明书附图元数据级问题**（如"说明书文字中提及的附图标记未在附图中出现"）：从 extracted_text 中定位说明书附图章节的实际文本（通常为末尾附近的 `"图1"`、`"图2"` 等），选择其中一个作为 context 和 highlight_text
  - **如果实在找不到任何可用的实际文本**（极端情况）：context 可设为文档中最接近的章节标题（如 `"附图说明"`），但必须在 issue 中明确说明问题详情

- **context 字段示例对比**：

| 规则 | 问题类型 | ❌ 错误context | ✅ 正确context |
|:-----|:---------|:--------------|:-------------|
| 规则3 | 摘要附图来源异常 | `摘要附图图1（源文件：image1.png）` | `图5`（摘要附图章节实际文本） |
| 规则4 | 附图标记缺失 | `说明书附图图1/图2/图3` | `图1`（说明书附图章节实际文本） |
| 规则1 | 摘要多张附图 | `摘要附图章节包含2张图片` | `图5`（摘要附图章节实际文本） |
| 规则5 | 实用新型缺附图 | `实用新型申请必须有附图` | `本实用新型`（说明书中的类型声明句） |
| 规则4(子) | 图片中罗马数字 | `说明书附图图3：附图标记'Ⅲ'` | `图3`（说明书附图章节对应图号） |
| 规则7 | 附图重复 | `说明书附图中image1重复出现` | `图2`（重复图片对应的图号） |
| 规则8 | 摘要附图无文字 | `摘要附图章节没有文字` | `摘要附图`（页眉章节标题）+ highlight_text="摘要附图" |
| 规则9 | 附图缺少名称 | `image3下方没有文字` | `图2`（相邻图的图号） |
| 规则10 | 编号不连续 | `附图编号1,2,4缺少3` | `图4`（编号跳跃处的图号） |
| 规则11 | 附图说明缺失 | `图3在附图说明中未提及` | `图3`（说明书附图章节实际文本） |
| 规则12 | 说明书附图无锚定文本 | `说明书附图章节没有文字` | `说明书附图`（页眉章节标题）+ highlight_text="说明书附图" |

- 对于图片级问题（规则4、6、11，来自图片子Agent），context 由图片子Agent按其规则格式提供（`<图片逻辑名称>：<位置描述>`），本Agent汇总时**必须逐条检查并修正**：如果发现context包含描述性文字、概括性表述或任何非文档原文的内容，必须立即修正为文档实际文本（如将描述性context替换为对应图号"图1"、"图2"等）。此步骤必须在保存最终JSON前执行，禁止跳过（依据SKILL.md约束#39）
